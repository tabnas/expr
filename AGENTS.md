# Agents Guide — expr

## What this project is

`@tabnas/expr` is a **Pratt expression-operator plugin** for the
[`tabnas`](https://github.com/tabnas/parser) parsing engine. It adds
infix, prefix, suffix, ternary, and paren operators with configurable
precedence on top of jsonic's relaxed-JSON grammar. Expressions parse
into LISP-style S-expressions — arrays whose first element is the
operator `Op` and whose remaining elements are the operand terms
(`1+2*3` → `[+, 1, [*, 2, 3]]`) — which a user-supplied evaluator can
reduce to values.

It is **not** standalone: like `@tabnas/csv` it layers on the
[`@tabnas/jsonic`](https://github.com/tabnas/jsonic) base grammar (the
relaxed-JSON `val`/`map`/`list`/`pair`/`elem` rules) and the engine's
lexer. The plugin uses `tn.rule(...)` (TS) / `j.Rule(...)` (Go) to wire
the operator-entry `expr` and grouping `paren` rules onto jsonic's
`val`, **not** the declarative `tn.grammar(...)` spec that `json`/`csv`
use.

There are two implementations that must behave identically — TypeScript
(canonical) and a Go port.

## The tabnas engine dependency

Both runtimes depend on the unpublished `@tabnas` siblings via a
**sibling checkout** (the standard tabnas dev model until the packages
publish tagged releases):

- TypeScript: `@tabnas/parser` and `@tabnas/jsonic` are declared as
  `peerDependencies` (`">=2"`) in `ts/package.json` and mirrored as
  `file:../../parser/ts` / `file:../../jsonic/ts` devDependencies for
  local builds. `@tabnas/debug` and `@tabnas/railroad` are dev-only
  `file:` devDependencies (debug for the `debug-model.test.ts`
  composition test, railroad to regenerate `ts/doc/grammar.{svg,txt}`).
  npm >=7 / Node >=24 auto-installs the peers; `engines.node` is `">=24"`.
- Go: `go/go.mod` requires `github.com/tabnas/jsonic/go` with
  `replace github.com/tabnas/jsonic/go => ../../jsonic/go`. That is the
  module's **only** tabnas dependency — the Go jsonic package re-exports
  the engine types (`jsonic.Make`, `jsonic.Rule`, `jsonic.Context`, …),
  so `go/expr.go` imports `jsonic`, not `parser`, directly.

Clone the siblings (`parser`, `jsonic`, plus `debug`/`railroad` for the
optional/diagram steps) next to this repo and build their TS first. CI
does this for you (see below).

## Authority and alignment rules

**TypeScript is canonical. Go is a port of it.** When you change
behaviour:

1. Change `ts/src/expr.ts` first.
2. Port the same change to `go/expr.go`.
3. Add/extend the shared fixture(s) in `test/spec/*.tsv` so both
   runtimes assert the new behaviour. The `.tsv` fixtures are the parity
   contract; both suites resolve them and both must stay green:
   - TS: `ts/test/spec-util.ts` `loadSpec` resolves `../../test/spec`
     from `dist-test/` (compiled tests run from `dist-test/`, specs live
     in repo-root `test/spec/`); `ts/test/spec.test.ts` `runSpec` drives
     them.
   - Go: `go/expr_test.go` `loadSpec`/`runSpec` resolve `../test/spec`
     relative to the test file; one `TestSpec*` func per fixture.
4. Mirror new unit cases across the TS suite and `go/expr_test.go`.

Several fixtures are run with **per-test custom operators** layered on
the defaults (e.g. `spec.test.ts` registers `@`, `~`, `!`, `?` with
specific binding powers for the unary-edge specs). When you add such a
fixture, register the **same** operators with the **same** powers in the
matching Go test, or the parity breaks.

## The binding-power ladder — only ORDER matters, magnitudes are volatile

This is the single most important thing to understand before touching
precedence. Operator precedence is encoded as integer binding powers in
`Expr.defaults.op` (TS, `ts/src/expr.ts` near the `Expr.defaults = {`
block) and the mirrored `defaults` map in `go/expr.go`'s `addDefaultOps`
(`makeAllOps` then builds the op list from those). The Pratt core
compares them only with `<` / `<=` — it never
depends on the *gap* between two numbers, only their **order**.

The defaults sit in a compact low block on a **base-1,000,000 ladder**,
leaving the whole range above the built-ins open for client operators:

```
   <1000000  looser client ops (assignment, ternary, logical, comparison)
    1000000  sequence / comma
    2000000  addition / subtraction                 <-- built-in
    3000000  multiplication / division / remainder  <-- built-in
    4000000  unary prefix  (+ -)                     <-- built-in (tightest)
    5000000+ free for tighter client ops (exponent, postfix, call/member)
```

Concretely the defaults are: `positive`/`negative` (prefix, `right:
4000000`, tightest); `multiplication`/`division`/`remainder` (infix,
`left: 3000000`, `right: 3100000`); `addition`/`subtraction` (infix,
`left: 2000000`, `right: 2100000`); plus the `plain` `(`…`)` paren op.

Two consequences agents must internalise:

- **`left < right` means left-associative.** `2+3+4 === (2+3)+4`
  because `addition.left (2000000) < addition.right (2100000)`. The
  `+100000` on `right` is the left-associativity offset; a tier occupies
  a 100000-wide band inside its 1000000-wide slot so adjacent tiers can
  never overlap. Swap them (`left > right`) for right-associative ops.
- **The magnitudes are free and volatile; only the order is a
  contract.** The base scale was deliberately rescaled (from an older
  ~1,000 scale to ~1,000,000) to give downstream clients headroom both
  above the tightest built-in and below addition. **Any future rescale
  must be an order-preserving remap of BOTH the defaults in
  `expr.ts`/`expr.go` AND every operator power baked into the tests**
  (`spec.test.ts` and `expr.test.ts` register ops like `right: 5000000`
  and `left: 7000000`; `ternary.test.ts` uses `left: 6000000`; the Go
  `expr_test.go` mirrors these and also registers `left: 15000000`/
  `right: 14000000`). Change one side's scale without
  remapping the other and the `.tsv` oracle silently stops testing what
  you think it does. To add a single operator instead, pick an unused
  tier base `N*1000000` (left = base, right = base+100000 for
  left-assoc) — no rescale needed.

## debug-model composition test (@tabnas/debug)

`ts/test/debug-model.test.ts` composes the expr plugin with the external
`@tabnas/debug` plugin and asserts the structured grammar model returned
by `debug.model()`:

- rule set `['elem','expr','list','map','pair','paren','val']` (the
  shared jsonic rules plus expr's own `expr` and `paren`);
- `m.config.start === 'val'` (note `m.config.start`, **not** `m.start`,
  which is undefined in this engine);
- `Expr` is in `m.plugins`;
- push edges: `val` pushes `expr`/`map`/`list`, `expr` pushes
  `paren`/`val`, `paren` pushes `val`.

It loads `@tabnas/debug` dynamically and **skips** unless the plugin is
resolvable — it is a `file:` devDependency so plain `npm test` runs it;
set `TABNAS_DEBUG_PATH` to point at a built sibling checkout otherwise.

`ts/test/doc-examples.test.ts` extracts and runs fenced assertion blocks
from the README/docs — keep doc examples correct.

## Build & test

TypeScript (from `ts/`):

```bash
npm install            # auto-installs peers; resolves file: siblings
npm run build          # tsc --build src test
npm test               # pretest builds, then node --test over dist-test/**/*.test.js
```

Go (from `go/`):

```bash
go build ./...
go test -v ./...       # unit tests + shared .tsv fixtures
```

The repo root [`Makefile`](Makefile) wraps both: `make build|test` run
the TS and Go halves, and `make publish-go V=x.y.z` injects `V` into the
`const Version` in `go/expr.go` (currently `0.1.3`) and tags
`go/vX.Y.Z`. The TS package version is `2.3.2` (`ts/package.json`).
Local builds resolve the unpublished siblings via the repo-set `go.work`
+ node_modules symlinks (`admin/scripts/link.sh`); there is no
checked-in `go.work` in this repo.

## CI

`.github/workflows/build.yml` has a **single `build` job** (no separate
build-go job — the Go port is exercised locally / via the Makefile, not
in CI):

- Matrix Ubuntu/Windows/macOS, Node 24.
- Sets `git config --global core.autocrlf false` (CRLF corrupts the
  `.tsv` fixtures).
- git-clones the tabnas closure (`parser debug json abnf railroad
  jsonic`) as sibling dirs, runs `npm i && npm run build --if-present`
  in each in topo order, then `npm test` in `expr/ts`.
